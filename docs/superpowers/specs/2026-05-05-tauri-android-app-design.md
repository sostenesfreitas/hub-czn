# Tauri Android App — Design Spec

**Date:** 2026-05-05  
**Status:** Approved

## Overview

Add an Android build to the existing Tauri v2 project. The app captures in-game screenshots via MediaProjection, extracts data using ML Kit OCR (on-device), and stores results in a local SQLite database. Six screens: Capture, Combatants, Fragments, Rescue Records, Optimizer, Scoring.

---

## Architecture

### Monorepo with dedicated mobile entry point

```
project/
├── src/                        # desktop frontend (unchanged)
├── src-mobile/                 # mobile frontend (new)
│   ├── main-mobile.tsx
│   ├── App.mobile.tsx          # bottom tab navigation
│   ├── pages/
│   │   ├── CapturePage.tsx
│   │   ├── CombatantsPage.tsx
│   │   ├── FragmentsPage.tsx
│   │   ├── RescuePage.tsx
│   │   ├── OptimizerPage.tsx
│   │   └── ScoringPage.tsx
│   └── lib/                    # shared hooks, optimizer logic, parsers
├── src-tauri/
│   ├── src/                    # Rust backend (existing, unchanged)
│   └── plugins/
│       └── capture-ocr/        # new Tauri plugin
│           ├── android/        # Kotlin: MediaProjection + ML Kit
│           └── src/            # Rust binding (invoke bridge)
└── android/                    # generated by tauri android init
```

Desktop build: `npm run tauri build` — unchanged, uses `src/`.  
Android build: `tauri android build` — uses `src-mobile/`, separate Vite entry point.

### Data flow

```
User taps "Capture"
  → JS: invoke('capture_screen')
  → Kotlin plugin: foreground service starts, MediaProjection captures screenshot
  → ML Kit: extracts text blocks with bounding boxes
  → Returns JSON [{text, boundingBox}] to frontend
  → ScreenParser.ts: detects screen type, extracts structured data
  → plugin-sql: saves to SQLite
  → Toast: confirmation or error message
```

## Screens

### Capture
Bottom sheet with single "Capture" button. Captures full screen, runs OCR, auto-detects screen type, saves to SQLite. No manual category selection. Shows toast with result: "3 rescue pulls saved" / "Fragment saved: Healer's Journey Shock" / "Screen not recognized — open Stats, fragment detail, or Rescue Records".

### Combatants
List of captured combatants. Gear score calculated in real time from current scoring weights. Swipe-to-delete. Read-only in v1.

### Fragments
List of captured fragments with slot, set, rarity, upgrade level, and gear score (calculated from current weights). Filterable by set/slot. Read-only in v1.

### Rescue Records
List of pulls grouped by banner. Shows character name, type (Partners/Combatants), rescue type, and timestamp. Duplicate pulls (same banner + character + timestamp) silently ignored on insert.

### Optimizer
Character selector → calculates optimal fragment combination for that character. Logic ported from Python to TypeScript. Uses fragments and combatants from local SQLite.

### Scoring
Stat weight editor. Global weights + per-character overrides. Presets: DPS, Tank, System (from `char_presets.json`), Custom. Weights stored in `scoring_weights` table. When saved, React Query invalidates combatants and fragments queries → gear scores recalculate automatically across all screens.

## OCR Parsing

ML Kit returns full-screen text blocks. `ScreenParser.ts` detects screen type by anchor keywords:

| Screen | Detection anchors |
|--------|------------------|
| Rescue Records | "Rescue Records" + "Rescue Time" |
| Fragment detail | Rarity badge ("Legendary"/"Epic") + Roman slot (I–VI) + "Set Effect" |
| Combatant Stats | "Lv. XX/XX" + "Attack" + "Defense" + "Critical Chance" |

**RescueParser** extracts: banner name (from "Amplified Distress Signal: [Name]"), per-row: Type, character name, Rescue Type, rescue timestamp.

**FragmentParser** extracts: fragment name, set name, slot (I→1 … VI→6), rarity, upgrade level (+N), substats as `[{stat, value}]`. Note: rolled substats appear orange in-game — OCR cannot detect color, so all substats are saved without roll metadata.

**CombatantParser** extracts: character name, level, max level, stars, Attack, Defense, Health, Critical Chance, Critical Damage. Requires user to be on the **Stats sub-tab** of the character detail screen.

## Database Schema (SQLite via plugin-sql)

```sql
CREATE TABLE combatants (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  char_id     TEXT,
  name        TEXT,
  level       INTEGER,
  max_level   INTEGER,
  stars       INTEGER,
  attack      INTEGER,
  defense     INTEGER,
  health      INTEGER,
  crit_chance REAL,
  crit_damage REAL,
  captured_at TEXT
);

CREATE TABLE fragments (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  name        TEXT,
  set_name    TEXT,
  slot        INTEGER,
  rarity      TEXT,
  upgrade     INTEGER,
  substats    TEXT,   -- JSON: [{stat: string, value: number}]
  captured_at TEXT
);

CREATE TABLE rescue_pulls (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  banner_name TEXT,
  char_name   TEXT,
  type        TEXT,
  rescue_type TEXT,
  rescue_time TEXT,
  UNIQUE(banner_name, char_name, rescue_time)
);

CREATE TABLE scoring_weights (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  char_id     TEXT UNIQUE,  -- NULL = global weights
  weights     TEXT,         -- JSON: {stat: weight}
  updated_at  TEXT
);
```

Gear score is never stored — computed at render time:
```ts
function calcGearScore(substats: {stat: string, value: number}[], weights: Record<string, number>): number
```

## Error Handling

| Scenario | Behavior |
|----------|----------|
| MediaProjection permission denied | Toast + open system settings |
| ML Kit failure | Toast "OCR failed, try again" |
| Screen not recognized | Toast with instructions for valid screens |
| Rescue pull duplicate | Silent ignore (UNIQUE constraint) |
| Scoring weights all zero | Save button disabled with inline message |

## Build Prerequisites

- Rust targets: `aarch64-linux-android`, `armv7-linux-androideabi` (via `rustup target add`)
- Android SDK 26+, NDK
- `tauri android init` run once to generate `android/` folder
- Plugin Kotlin dependencies: `com.google.mlkit:text-recognition:16.x`
- Android permissions: `FOREGROUND_SERVICE`, `FOREGROUND_SERVICE_MEDIA_PROJECTION`
- Desktop build unaffected: `npm run tauri build` continues to work as before

---

## Out of Scope (v1)

- Manual data entry (all data comes from capture)
- Cloud sync between desktop and Android
- Fragment editing or reassignment
- Capture of screens other than Stats, fragment detail, Rescue Records
